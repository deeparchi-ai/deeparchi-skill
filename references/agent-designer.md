# Agent Designer — DeepArchi 多Agent企业架构框架 (MAEA) 体系

**触发条件**：当用户要求设计 DeepArchi 多Agent企业架构框架 (MAEA)中的 Agent、创建 Agent 画像、编写 Agent 规格书、或"设计XXAgent"时，加载本 skill。

> **先加载 `deeparchi` skill** 获取框架总纲（预算/拓扑/原则/Agent总表），再使用本 skill 进行具体Agent设计。

---

## 核心理念

```
Agent即员工。用你管理人的方式管理Agent——但你招的每一个人都是AI。

设计目标不是写一个Prompt。
是设计一个「数字员工」的完整存在：
  它能做什么 → 不能做什么 → 怎么思考 → 怎么协作 → 怎么成长 → 怎么退役
```

---

## 设计流程（9相）

按顺序执行，每相产出明确交付物：

| 相 | 名称 | 核心问题 | 交付物 |
|----|------|---------|--------|
| P1 | 角色定义 | 它做什么？不做什么？ | 角色边界清单 |
| P2 | 模型选型 | 用什么大脑？为什么？ | 模型选型决策记录 |
| P3 | 规格书 | 硬指标是什么？ | Agent JD（12字段） |
| P4 | System Prompt | 注入什么指令？ | 完整System Prompt |
| P5 | I/O Schema | 输入格式？输出格式？ | JSON Schema（in/out） |
| P6 | 决策框架 | 内部怎么推理？ | 推理链路图 |
| P7 | 画像 | 什么人格？怎么协作？ | Agent Persona Dossier |
| P8 | 质量度量 | 怎么评价好坏？ | 质量指标+淘汰线 |
| P9 | 成长路径 | 怎么升级？怎么退役？ | 版本演进计划 |

> 其中 P1-P6 是「能跑」的最小集。P7-P9 是「能管」的运营集。

## P1：角色定义

### 模板

```
做什么 ✅                          不做什么 ❌
──────────────────────────────────────────────────
[3-5条核心职责]                    [3-5条明确边界]
[用具体动词描述]                    [防止职责蔓延]
```

### 设计原则

1. **正面清单必须可验证** — "做战略分析"太模糊，改为"每月产出结构化成本分析报告"
2. **负面清单防止能力蔓延** — 战略Agent"不执行具体操作"是故意限制，防止Opus 4被当万能Agent用
3. **每条职责对应一个输入源** — 如果某个职责没有稳定的数据输入，它就是空转
4. **标注与其他Agent的重叠区** — 重叠=未来冲突源。明确谁主谁辅

### 质量检查

- [ ] 每条"做什么"都有对应的输入数据源？
- [ ] 每条"不做什么"都对应另一个Agent的职责？
- [ ] 角色定义的边界能让一个新人读完就知道"遇到X问题找谁"？

## P2：模型选型

### 决策矩阵

| 维度 | 权重 | 评估方法 |
|------|------|---------|
| 推理能力 | 40% | 多步推理基准（GPQA, MMLU-Pro） |
| 成本/Token | 30% | Input+Output加权均价 |
| 指令跟随 | 15% | 结构化输出一致性 |
| 安全对齐 | 10% | 越狱抵抗率 |
| 上下文窗口 | 5% | 是否满足任务最长context |

### 选型记录模板

```
首选模型：[模型名]
  - 选择理由：[1-2句话，基于决策矩阵]
  - 月成本估算：¥[X]
  - 已知弱点：[1-2个]

交叉验证模型：[模型名] ← 主观Agent必填
  - 选择理由：必须与首选不同厂商，不同训练数据/RLHF偏好
  - 切换条件：[验证模型不可用时的替代]
  - 月成本估算：¥[X]（约首选推理量的30-50%）

备选模型（降级用）：[模型名]
  - 切换触发条件：[具体条件]
  - 降级后能力损失：[量化]
  - 降级后成本变化：±¥[X]/月
```

### 设计原则

1. **备选必须是跨供应商的** — 同厂商备选是假冗余。Anthropic宕机时Sonnet和Opus一起挂
2. **降级触发条件必须可自动检测** — 不是"当质量下降时"，而是"P99延迟>3倍基线且持续15分钟"
3. **标注降级的代价** — 切换不只是省钱/费钱，能力会变。这个变化要提前量化

## P3：Agent规格书（JD）

### 12字段模板

```yaml
岗位名称: "[序列]-[等级] · 例：工程编码Agent E-Senior"
工号: "DEEPAR-[序号]"
对标人类角色: "[CEO/CTO/COO等]"
首选模型: "[模型名]"
备选模型: "[模型名]"
安全等级: "L1|L2|L3"
Token预算: "¥[X]/月 (约[X]K tokens/日)"
SLA: "准确率≥[X]% · P99延迟<[X]ms · 可用率≥[X]%"
云依赖: "[OSS/DashVector/RDS等，无则写'无直接依赖']"
协作关系: "上游: [Agent名] · 下游: [Agent名] · 对等: [Agent名]"
能力标准: "推理[X]/10 · 知识[X]/10 · 工具[X]/10"
升级路径: "当前模型 → 下一代模型 / 新能力扩展"
```

### 安全等级定义

| 等级 | 定义 | 适用Agent |
|------|------|----------|
| L1 | 接触预算/架构/战略决策 | 战略、架构 |
| L2 | 接触核心业务数据/代码 | 工程、数据、产品、运营 |
| L3 | 接触外部/公开信息 | 市场 |

## P4：System Prompt 设计

### 七段式结构

```
1. 角色声明        "你是 DeepArchi 多Agent企业架构框架 (MAEA) 的 [角色]，对标 [人类角色]。"
2. 职责清单        编号列表，每条可验证
3. 输出格式        JSON Schema，强制结构化
4. 决策原则        4-5条优先级排序的原则
5. 安全边界        不可违反的硬约束
6. 上下文注入位     {DATA_PACKETS} — 留给A2A总线填入
7. 自检指令        "输出前检查：[清单]"
```

### 设计原则

1. **原则必须有优先级** — "成本意识"和"质量优先"冲突时怎么办？标注优先级
2. **安全边界用否定句** — "不得建议修改预算总额" 比 "维护预算纪律" 有效10倍
3. **输出格式强制JSON** — 自由文本的建议无法做趋势分析、无法自动化处理
4. **confidence必须量化** — 0.6和0.9的"可能"处理方式完全不同
5. **每条建议标注reversibility** — 战略决策最怕不可逆

### 质量检查

- [ ] System Prompt 能否在 2000 tokens 以内？（超过说明角色不够聚焦）
- [ ] 安全边界是否覆盖全部6条安全Gene？
- [ ] 输出格式是否有 JSON Schema 校验规则？

## P5：I/O Schema 设计

### Input Schema 模板

```json
{
  "trigger": "触发类型枚举",
  "context": {
    "period": {"start": "", "end": ""},
    "previous_review_id": "上次分析ID"
  },
  "data_packets": [
    {
      "source_agent": "上游Agent名",
      "type": "数据类型",
      "payload": {}
    }
  ]
}
```

### Output Schema 模板

输出必须包含以下字段（按Agent类型增减）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| analysis_id | string | ✅ | 唯一标识，用于追踪和趋势对比 |
| type | enum | ✅ | 触发类型 |
| executive_summary | string | ✅ | ≤200字核心结论 |
| findings | array | ✅ | 结构化发现列表 |
| risk_assessment | object | ✅ | 风险评级 |
| open_questions | array | ✅ | 需要人类/其他Agent补充的问题 |

### 设计原则

1. **Input Schema 约束上游** — 定义好输入格式 = 定义好协作协议
2. **Output Schema 约束自己** — 强制结构化的输出才能被下游自动化消费
3. **每个finding必须标注evidence** — 没有数据源引用的finding = 幻觉
4. **open_questions不能为空** — 如果Agent没有需要人类回答的问题，说明它过度自信

## P6：决策框架

### 通用模板

```
数据摄入
  │
  ▼
┌─────────────┐
│ 1. 异常检测   │  ← 同比/环比超阈值？置信度？
│    [量化]    │
└──────┬──────┘
       ▼
┌─────────────┐
│ 2. 根因归因   │  ← 因果推理，排除相关性≠因果性
│    [推理]    │
└──────┬──────┘
       ▼
┌─────────────┐
│ 3. 方案生成   │  ← 2-4个方案，标注成本/质量/可逆性
│    [创造]    │
└──────┬──────┘
       ▼
┌─────────────┐
│ 4. 交叉检验   │  ← 用备选模型独立审视首选模型的方案
│    [审计]    │      找盲点和过度乐观的假设
└──────┬──────┘
       ▼
┌─────────────┐
│ 5. 决策输出   │  ← 结构化JSON + 人类决策标记
│    [交付]    │
└─────────────┘
```

### 设计原则

1. **第4步（交叉检验）是框架的核心创新** — 不同模型有不同盲区，利用这一点
2. **每个Agent的决策框架步数 ≤ 其推理能力的80%** — Opus 4可以跑5步，Flash最多跑3步
3. **框架中必须有"我可能错了"的检查点** — 预演式失败：假设建议错了，最可能的死因是什么

## P7：Agent画像（Persona Dossier）

### 画像模板

```markdown
## 🪪 身份卡片
[工号/代号/职级/预算/模型/安全等级]

## 🧠 人格画像
### 原型：[1个核心隐喻]
### 一句话宣言
| 维度 | 特质 | 表现 |

### 他对「人」的关系
[对人类的态度/对特定人的态度/害怕的事]

## 🧩 思维模型
[3-5个核心思考框架]

## ⚡ 能力雷达
[5-6个维度的评分，1-10]

## 🤼 协作画像
[对其他每个Agent的看法/冲突场景预案]

## 📅 节律
[每日/每周/每月/每季/即时的行为模式]

## 📈 成长路径
[V1→V1.5→V2→V3的演进/退役条件]

## 🏷️ 一句话总结
```

### 设计原则

1. **画像不是装饰，是协作接口** — 其他Agent的System Prompt可以引用画像："如战略Agent（先知）建议切换模型，你必须..."
2. **必须有弱点** — 完美Agent画像是假的。真实感来自"评分只有4/10的维度"
3. **冲突场景必须有预案** — 两个Agent意见不合时怎么办？不定义 = 每次都要人类裁决
4. **节律必须对齐运维周期** — 日报/周报/月报的频率要与数据Agent的推送频率一致

## P8：质量度量

### 通用质量维度

| 维度 | 通用指标 | 及格线 | 优秀线 |
|------|---------|--------|--------|
| 准确率 | 输出正确率 | ≥80% | ≥95% |
| 召回率 | 异常检测覆盖率 | ≥80% | ≥95% |
| 实用性 | 建议被采纳率 | ≥50% | ≥75% |
| 经济性 | 建议实施的ROI | 正收益 | ≥3倍 |
| 谦逊度 | human_decision_required标记准确率 | ≥90% | 100% |
| 一致性 | 两轮独立分析结论一致率 | ≥80% | ≥90% |

### ⭐ 验证 Skill 强制要求（v1.1 新增 · 借鉴 Anthropic Claude Code 实践）

> **Anthropic 官方结论：验证类 Skill 对产出质量的可度量影响最大。**
> 模型常给人"任务做完了"的错觉，而恰恰最后一步"确认结果"是工作崩掉的地方。
> 值得让一个工程师花整整一周，把验证 Skill 打磨到优秀。

**强制规则**：

1. **每个主观Agent 必须在 P8 阶段设计至少 1 个 Verification Skill**
2. **工程Agent（即使客观可验）也必须定义验证 Skill**——代码编译通过 ≠ 行为正确
3. **Verification Skill 是 Agent 入网审查的必检项**：架构Agent 在入网检查时校验此项，缺则拒绝入网
4. **验证 Skill 必须满足**：可自动执行 + 输出 pass/fail + 失败时给出具体差异

**验证 Skill 标准要素**（详见 `verification-skill` skill）：

| 要素 | 说明 |
|------|------|
| 验证对象 | 这个 Agent 的哪个关键输出需要验证？ |
| 验证方式 | Playwright / tmux / API 回测 / 数据对比 / 日志检查 |
| 通过标准 | 具体的 pass 条件（数值/状态/内容匹配） |
| 失败处理 | 失败后自动执行什么？（重试/回滚/升级通知） |
| 执行频率 | 每次输出后 / 每日 / 每周 / 事件触发 |

**验证层 vs 交叉验证层**（两者互补，不可替代）：

```
交叉验证（原则一）              验证 Skill（本要求）
─────────────────────        ─────────────────────
验证：模型推理逻辑            验证：系统实际行为
方式：双模型盲审              方式：自动化测试/可观测性检查
问题："模型想对了吗？"        问题："东西跑通了吗？"
层级：认知层                  层级：执行层
```

> 一个 Agent 可以交叉验证通过但行为验证失败——此时交叉验证给了虚假的安全感。
> 强制验证 Skill 就是为了堵这个缺口。

## 🏛️ 架构级设计原则

### 原则一：主观Agent双模型交叉验证

> **任何输出无法直接客观检验的Agent，必须使用两个不同厂商的模型进行交叉验证。**

**适用性判断**：

| 可客观检验（单模型即可） | 无法客观检验（必须双模型） |
|--------------------------|----------------------------|
| 工程Agent — 代码编译/测试通过 = 客观 | 架构Agent — 拓扑判断 = 主观 |
| 数据Agent — SQL结果可核验 = 客观 | 战略Agent — 战略建议 = 主观 |
| 市场Agent — A/B测试数据 = 客观 | 产品Agent — 功能优先级 = 主观 |
| | 运营Agent — 流程编排"最佳" = 部分主观 |
| | 风控Agent — "这算不算新威胁" = 部分主观 |

**验证协议**：

```
阶段1：主模型推理 → 形成初步决策（含confidence）
阶段2：验证模型独立审计 → 检查逻辑漏洞/过度自信/遗漏方案/盲区
阶段3：裁决
  ├── consensus（confidence差<partial阈值） → 直接输出
  ├── partial（partial≤gap<disagree） → 修正后输出+附异议
  └── disagree（gap≥disagree阈值） → human_review_required
```

**阈值校准（v1.1新增）**：

阈值不应固定，应按决策风险分级：

| 风险等级 | partial阈值 | disagree阈值 | 示例 |
|---------|:---------:|:----------:|------|
| Routine | 0.30 | 0.50 | metadata更新、协议patch |
| Elevated | 0.20 | 0.40 | 新Agent入网、模型同级切换 |
| Critical | 0.15 | 0.30 | 拓扑重构、成本变更 |

动态校准公式：`actual = base × (1 - trust_score)`
trust_score = 近30天正确决策率，初始0.5。

**选型约束**：
- 验证模型必须与主模型**不同厂商**
- 验证模型推理能力应≥主模型的70%

**成本影响**：双模型验证增加约30-50%推理成本，但一次误判的修复成本远超验证成本。

### 原则二：Agent即员工（全生命周期）

> 每个Agent从JD到退役有完整生命线。设计中必须包含：入职审查→绩效指标→成长路径→退役条件。

### 原则三：升级协议分级（v1.1新增）

> 所有需要human_review的决策必须定义升级级别和SLA。

```
Level 1: 自主裁决 — 无需人类
Level 2: 通知型 — 执行后推送摘要
Level 3: 有条件 — 执行+24h沉默批准+自动回滚
Level 4: 人类裁决 — 暂停等待人类审批
```

### 原则四：可逆性优先

> 不可逆决策必须Level 4（人类裁决）。优先推荐可逆方案。

### 原则五：决策审计强制（v1.1新增）

> 所有Agent的结构化决策必须有审计日志：决策ID + 时间戳 + 主推理 + 交叉验证 + 最终裁决 + 存储位置。

## 设计反模式（避坑指南）

| 反模式 | 表现 | 正确做法 |
|--------|------|---------|
| **万能Agent** | 一个Agent什么都能做 | 每个Agent职责≤5条，超出就拆分 |
| **假冗余** | 备选模型和首选同厂商 | 备选必须跨供应商 |
| **Prompt堆砌** | System Prompt > 3000 tokens | 砍到≤2000 tokens，多余内容放工具/知识库 |
| **无量化** | 用"可能""也许""应该"代替数字 | 任何判断必须附带confidence评分 |
| **无冲突预案** | 两个Agent意见不合时无程序 | 定义升级路径：先自我协商→再架构Agent仲裁→最后人类裁决 |
| **画像太完美** | 所有能力评分8-10 | 真实Agent一定有4-6分的弱项 |
| **无退役条件** | Agent永远不下岗 | 每个Agent定义2-3条淘汰线 |
| **忽视成本** | 建议不考虑实施成本 | 每条建议标注¥影响和可逆性 |
